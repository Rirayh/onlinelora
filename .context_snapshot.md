Adapter wrapped successfully (896 LoRA layers, active_adapters=['default']) but produces ZERO effect on logits because lora_B is all-zero matrix → A·0 = 0 increment.

### Why
`merge_and_reset_lora` in `scripts/stage3_run.py` resets B=0 after each merge event. If training ends ON OR after a merge boundary (with no further B accumulation), the saved adapter is empty residual.
- `relora_baseline` (merge_every=500, total=3000): merges at 500/1000/1500/2000/2500/3000 → final adapter is the post-step-3000 reset = **B=0**
- Same for S3pos, S3neg, random_drop, train_gated, cola

### IMPACT
**ALL "best results" we've been celebrating for merge-based methods are actually the BASE MODEL scores.**
- qwen3-8b/tulu/S3pos best 87.04% / step_3000 88.32% / step_1500 88.48% — all = qwen3-8b base GSM8K
- qwen3-8b/mm/S3pos 86.13% — also possibly base
- All olmo2 contamination is the same bug
- qwen25, mistral, all other merge-method results — all bogus

### What IS correct
- lora_vanilla cells (no merge → B retains training values)
- dora cells (different update scheme; need to verify but probably OK)
- adalora cells (different scheme)

### The fix
Two options:
(a) **Don't reset adapter at training end**: track `adapter_at_end_of_training` separately; before saving, snapshot the post-last-train-step adapter (BEFORE the final scheduled merge). Implement in stage3_run.py.
(b) **Save merged base model**: after final merge, the base model has all updates. Save the FULL merged model (or save the cumulative ΔW = merged_base - original_base as a "rank-full pseudo-adapter"). For lm-eval, load that merged model directly.

Recommendation: (b) — cleaner for downstream evals.
